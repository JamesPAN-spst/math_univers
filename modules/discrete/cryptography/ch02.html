01块加密与 Feistel 网络

🎯 为什么研究这个问题

OTP 的密钥长度问题迫使我们转向计算安全性。块加密将固定长度的明文 块用短密钥加密为等长密文块，是现代对称加密的核心。Feistel 网络 提供了一种通用框架：用简单的轮函数迭代构造复杂的置换，DES 正是 其成功范例。理解 Feistel 结构是掌握现代块加密的起点。

🔍 抽象化方法

块加密 \(E_k: \{0,1\}^n \to \{0,1\}^n\) 是密钥索引的置换族。 Feistel 网络将非可逆函数 \(f\) 转化为可逆置换——这是一种巧妙的 结构性保证可逆性的方法。

📐 公式推导 / 模型建立

Feistel 网络：将 \(2n\) 位输入分为 \((L_0, R_0)\)， 每轮 \(i\) 的变换： \[L_i = R_{i-1}, \quad R_i = L_{i-1} \oplus f(R_{i-1}, K_i)\] 其中 \(f\) 为轮函数，\(K_i\) 为子密钥。 解密完全对称：\(R_{i-1} = L_i\)，\(L_{i-1} = R_i \oplus f(L_i, K_i)\)。 关键性质：无论 \(f\) 是否可逆，整体变换都可逆。 DES：64 位块、56 位密钥、16 轮 Feistel。轮函数包含 扩展置换(E)、S 盒替代、P 置换。 弱点：密钥仅 56 位，\(2^{56} \approx 7.2 \times 10^{16}\)， 已被专用硬件暴力破解。3DES 用 \(E_{k_1}(D_{k_2}(E_{k_1}(m)))\) 将有效密钥长度提升到 112 位。

🌐 实际应用

- DES 曾是 NIST 标准（1977-2005），广泛用于银行和通信 - Feistel 结构影响了 Blowfish、Camellia 等后续设计

✏️ 例题分析与易错点

例题：一轮 Feistel 中 \(L_0=1010\)，\(R_0=0110\)，\(f(R_0,K)=1100\)。 求 \((L_1, R_1)\)。 解答：\(L_1 = R_0 = 0110\)；\(R_1 = L_0 \oplus f = 1010 \oplus 1100 = 0110\)。 易错点：解密时子密钥顺序必须反转（\(K_{16}, K_{15}, \ldots, K_1\)）。 若用加密顺序解密，结果将完全错误。

02AES 算法的结构

🎯 为什么研究这个问题

DES 密钥过短促使 NIST 在 2001 年选定 Rijndael 算法作为 AES—— 当今最广泛使用的对称加密标准。AES 不采用 Feistel 结构，而是 使用代换-置换网络(SPN)，每轮对整个状态矩阵进行变换。 理解 AES 的四步轮变换是掌握现代加密实践的核心。

🔍 抽象化方法

AES 将 128 位明文视为 \(4 \times 4\) 的 \(GF(2^8)\) 矩阵， 每轮通过字节替代(非线性)、行移位(扩散)、列混合(扩散)、 密钥加(密钥引入)四步实现 confusion 与 diffusion。

📐 公式推导 / 模型建立

AES-128：128 位块、128/192/256 位密钥、10/12/14 轮。 状态矩阵 \(S = (s_{i,j}) \in GF(2^8)^{4\times 4}\)。 每轮四步操作： (1) SubBytes：每字节通过 S 盒替代（\(GF(2^8)\) 中取逆再仿射变换） \[s'_{i,j} = \text{S-box}(s_{i,j})\] (2) ShiftRows：第 \(i\) 行左移 \(i\) 个字节 (3) MixColumns：每列乘以 \(GF(2^8)\) 上的固定矩阵 \[\begin{pmatrix}2&3&1&1\\1&2&3&1\\1&1&2&3\\3&1&1&2\end{pmatrix} \begin{pmatrix}s_0\\s_1\\s_2\\s_3\end{pmatrix}\] (4) AddRoundKey：状态与轮密钥 XOR 安全性：10 轮后实现了充分的 confusion（S 盒非线性度高） 和 diffusion（4 轮后每个输出位依赖全部 128 个输入位）。

🌐 实际应用

- TLS/HTTPS、Wi-Fi (WPA2/3)、磁盘加密 (BitLocker) 均使用 AES - 现代 CPU 内置 AES-NI 指令实现硬件加速

✏️ 例题分析与易错点

例题：AES 的 SubBytes 步骤中，字节 \(0x53\) 的 S 盒输出是什么？ 解答：查 AES S 盒表，行 \(5\) 列 \(3\)：\(\text{S-box}(0x53) = 0xED\)。 （S 盒通过 \(GF(2^8)\) 求逆构造，无需记忆，需理解原理） 易错点：MixColumns 中的乘法在 \(GF(2^8)\) 上进行（模 \(x^8+x^4+x^3+x+1\)），不是普通整数乘法。

03块加密操作模式

🎯 为什么研究这个问题

块加密本身只能加密固定长度的块（如128位），但实际消息长度任意。 操作模式定义了如何将块加密扩展为任意长度消息的加密机制。不同 模式在安全性、并行性、错误传播方面各有优劣。ECB 的经典教训—— 企鹅图像加密后仍可辨认——生动展示了模式选择的重要性。

📐 公式推导 / 模型建立

设块加密 \(E_k\)，明文分块 \(m_1, m_2, \ldots\) ECB（电子密码本）：\(c_i = E_k(m_i)\)。 致命缺陷：相同明文块 → 相同密文块，泄露模式信息。 CBC（密码块链接）：\(c_i = E_k(m_i \oplus c_{i-1})\)，\(c_0 = \text{IV}\)。 解密：\(m_i = D_k(c_i) \oplus c_{i-1}\)。 优点：相同明文块因不同上下文产生不同密文。 缺点：加密不可并行，IV 必须随机且不重复。 CTR（计数器模式）： \[c_i = m_i \oplus E_k(\text{nonce} \| i)\] 将块加密转化为流加密，加密解密均可并行。 GCM（Galois/Counter Mode）：CTR 加密 + GHASH 认证， 提供认证加密(AEAD)——同时保证机密性和完整性。

🌐 实际应用

- TLS 1.3 推荐 AES-GCM 和 ChaCha20-Poly1305 - 磁盘加密使用 XTS 模式（可随机访问）

✏️ 例题分析与易错点

例题：CBC 模式中，若 \(c_2\) 传输中被篡改了一个比特， 哪些明文块的解密会出错？ 解答：\(m_2 = D_k(c_2) \oplus c_1\) —— 完全乱码（\(c_2\) 被改）。 \(m_3 = D_k(c_3) \oplus c_2\) —— 恰好在对应位翻转（\(c_2\) 影响 XOR）。 \(m_4\) 及之后不受影响。 易错点：认为 CBC 中一个错误会影响后续所有块。实际上错误只 传播到下一个块，因为 \(c_3\) 本身未被修改。

04流加密与消息认证码

🎯 为什么研究这个问题

并非所有场景都适合块加密——实时通信需要逐比特/逐字节加密的 流加密。此外，加密只保证机密性，不保证完整性：敌手可能篡改 密文。消息认证码(MAC)解决了这一问题，保证消息未被篡改。 「先加密再 MAC」还是「先 MAC 再加密」？这个顺序问题曾导致 多个实际系统的安全漏洞。

🔍 抽象化方法

流加密：由密钥生成伪随机流 \(z_1,z_2,\ldots\)，\(c_i = m_i \oplus z_i\)。 MAC：\(\text{Tag} = \text{MAC}_k(m)\)，验证 \((m, \text{Tag})\) 是否配对。

📐 公式推导 / 模型建立

流加密：核心是伪随机数发生器(PRG) \(G: \{0,1\}^s \to \{0,1\}^n\) 将短种子（密钥）扩展为长密钥流。 安全性要求：\(G(k)\) 与真随机序列不可区分。 代表算法：RC4（已不安全）、ChaCha20（现代推荐）。 消息认证码(MAC)： \(\text{Mac}: \mathcal{K} \times \mathcal{M} \to \mathcal{T}\)， 安全性：不可伪造性——敌手未知 \(k\) 时不能产生有效的 \((m, t)\) 对。 HMAC 构造：基于哈希函数 \(H\)， \[\text{HMAC}_k(m) = H\bigl((k \oplus \text{opad}) \| H((k \oplus \text{ipad}) \| m)\bigr)\] 可证明安全：若 \(H\) 的压缩函数是 PRF，则 HMAC 是安全 MAC。 认证加密(AEAD)：Encrypt-then-MAC（先加密再 MAC）是可证明安全的 组合方式；MAC-then-Encrypt 存在 padding oracle 攻击风险。

🌐 实际应用

- ChaCha20-Poly1305 被 TLS 1.3 和 WireGuard VPN 采用 - HMAC-SHA256 广泛用于 API 认证和 JWT 令牌

✏️ 例题分析与易错点

例题：为什么 CTR 模式加密后不加 MAC 是危险的？ 解答：CTR 中 \(c_i = m_i \oplus E_k(\text{nonce}\|i)\)。攻击者 翻转 \(c_i\) 的第 \(j\) 位，解密后 \(m_i\) 的第 \(j\) 位也翻转。 敌手可精确控制明文变化而不被检测。 易错点：认为「加密了就安全了」。加密保机密性不保完整性， 必须配合 MAC 或使用 AEAD 模式。
