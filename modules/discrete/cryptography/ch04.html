数字签名

01数字签名的概念与安全模型

🎯 为什么研究这个问题

在数字世界中，如何证明一条消息确实由某人发出且未被篡改？数字签名 是手写签名的电子对应物，具有不可伪造性和不可否认性。它是电子商务、 软件分发、区块链的安全基石。理解签名的安全定义（EUF-CMA）是 理解具体方案安全性论证的前提。

🔍 抽象化方法

签名方案 = \((\text{KeyGen}, \text{Sign}, \text{Verify})\) 三元组。 安全性定义为存在性不可伪造性——即使攻击者可获得任意消息的签名， 也无法伪造新消息的有效签名。

📐 公式推导 / 模型建立

数字签名方案： - \(\text{KeyGen}() \to (pk, sk)\) - \(\text{Sign}(sk, m) \to \sigma\) - \(\text{Verify}(pk, m, \sigma) \to \{\text{accept}, \text{reject}\}\) 正确性：\(\text{Verify}(pk, m, \text{Sign}(sk, m)) = \text{accept}\)。 EUF-CMA 安全（选择消息攻击下的存在性不可伪造性）： 攻击者可查询签名预言机 \(\text{Sign}(sk, \cdot)\)， 需伪造一对 \((m^*, \sigma^*)\) 其中 \(m^*\) 未被查询过。 \[\Pr[\text{Verify}(pk, m^*, \sigma^*) = \text{accept}] \leq \text{negl}(n)\] 哈希-然后-签名范式：先哈希消息 \(h = H(m)\)，再签名 \(h\)。 减小签名的计算量，且允许签名固定长度的数据。

🌐 实际应用

- 代码签名：操作系统验证驱动程序和应用的完整性 - TLS 证书链：CA 用数字签名为网站身份背书

✏️ 例题分析与易错点

例题：Alice 对消息 \(m\) 签名，Bob 验证。若 Eve 篡改为 \(m'\)， 原签名 \(\sigma\) 能否通过验证？ 解答：\(\text{Verify}(pk, m', \sigma) = \text{reject}\)，因哈希 \(H(m') \neq H(m)\)（碰撞抗性保证），签名验证失败。 易错点：数字签名 ≠ 加密。签名不保密消息内容，仅保证 完整性和来源认证。

02RSA 签名与 ElGamal/DSA 签名

🎯 为什么研究这个问题

RSA 和 ElGamal 不仅能加密，也能签名。RSA 签名简单直观—— 「用私钥加密」——但需要配合哈希和填充。ElGamal 签名衍生出 DSA（数字签名标准），长期作为美国 NIST 标准。理解这两类签名 是学习 ECDSA 的基础。

📐 公式推导 / 模型建立

RSA 签名：\(\sigma = H(m)^d \bmod n\)， 验证：\(\sigma^e \bmod n \stackrel{?}{=} H(m)\)。 安全性依赖 RSA 问题 + 哈希的碰撞抗性。 实际使用 PSS 填充（可证明安全）： \[\sigma = (\text{pad}(H(m), r))^d \bmod n\] DSA 签名：参数 \((p, q, g)\)，\(q | (p-1)\)，\(g\) 的阶为 \(q\)。 私钥 \(x\)，公钥 \(y = g^x \bmod p\)。 签名：随机 \(k\)， \[r = (g^k \bmod p) \bmod q, \quad s = k^{-1}(H(m) + xr) \bmod q\] 验证：\(w=s^{-1}\)，\(u_1=H(m)w\)，\(u_2=rw\)， 检查 \((g^{u_1}y^{u_2} \bmod p) \bmod q \stackrel{?}{=} r\)。

🌐 实际应用

- RSA-PSS 签名用于 TLS、PDF 文档签名 - DSA 是 FIPS 186 标准（2023 年起不再推荐新应用）

✏️ 例题分析与易错点

例题：RSA 签名中为什么必须先哈希再签名？ 解答：若直接签名 \(\sigma = m^d\)，攻击者可选 \(\sigma\) 计算 \(m = \sigma^e\) 获得有效的 \((m, \sigma)\) 对（存在性伪造）。 哈希后 \(m\) 无法被这样「逆向构造」。 易错点：DSA 签名中随机数 \(k\) 必须保密且不重复。 Sony PS3 私钥泄露事件就是因为 ECDSA 中重用了 \(k\)。

03椭圆曲线密码学基础

🎯 为什么研究这个问题

RSA 和 DLP 的安全性随计算力增长需要不断增大密钥。椭圆曲线 密码学(ECC)在更小的密钥长度下提供相同安全级别：256 位 ECC ≈ 3072 位 RSA。ECC 是移动设备和 IoT 的理想选择。理解椭圆 曲线上的群律是掌握 ECDH 和 ECDSA 的基础。

🔍 抽象化方法

椭圆曲线上的点构成阿贝尔群，群运算由切线法和割线法几何定义。 ECDLP（椭圆曲线离散对数）比普通 DLP 更难——没有亚指数算法。

📐 公式推导 / 模型建立

椭圆曲线（Weierstrass 形式）：\(y^2 = x^3 + ax + b\)（\(\Delta \neq 0\)）。 群律：点集 \(E(\mathbb{F}_p) \cup \{\mathcal{O}\}\)， \(\mathcal{O}\) 为无穷远点（单位元）。 - 逆元：\(-P = (x, -y)\) - 点加：\(P + Q\)（\(P \neq Q\)），过 \(P, Q\) 的直线交曲线于 \(R'\)， \(P + Q = -R'\)。斜率 \(\lambda = \frac{y_Q - y_P}{x_Q - x_P}\)。 - 倍点：\(2P\)，用切线。\(\lambda = \frac{3x_P^2 + a}{2y_P}\)。 - 结果：\(x_R = \lambda^2 - x_P - x_Q\)，\(y_R = \lambda(x_P - x_R) - y_P\)。 ECDLP：已知 \(P\) 和 \(Q = nP\)（\(n\) 次标量乘），求 \(n\)。 安全级别：256 位曲线提供 128 位安全性（最优攻击 \(O(\sqrt{n})\)， 即 Pollard rho），对应 RSA-3072。

🌐 实际应用

- Bitcoin 使用 secp256k1 曲线 - TLS 1.3 默认使用 X25519（Curve25519 上的 ECDH）

✏️ 例题分析与易错点

例题：在 \(E: y^2 = x^3 + 2x + 3\) over \(\mathbb{F}_5\) 上， 求 \(P=(1,4)\) 的 \(2P\)。 解答：\(\lambda = (3 \cdot 1 + 2)/(2 \cdot 4) = 5/8 \bmod 5\)。 \(5 \bmod 5 = 0\)，\(\lambda = 0\)。 \(x_R = 0 - 1 - 1 = -2 = 3 \bmod 5\)。 \(y_R = 0 \cdot (1-3) - 4 = -4 = 1 \bmod 5\)。\(2P = (3, 1)\)。 易错点：所有运算在 \(\mathbb{F}_p\) 上进行，除法是乘逆元。

04ECDSA 签名算法

🎯 为什么研究这个问题

ECDSA 将 DSA 迁移到椭圆曲线上，在保持安全性的同时大幅缩短 签名和密钥长度。它是 Bitcoin 和以太坊交易签名、TLS 证书、 物联网设备认证的核心算法。EdDSA（如 Ed25519）进一步简化了 实现并消除了随机数相关的安全隐患。

📐 公式推导 / 模型建立

ECDSA 签名：曲线 \(E\)，基点 \(G\)（阶 \(n\)），私钥 \(d\)，公钥 \(Q=dG\)。 签名消息 \(m\)： (1) 随机选 \(k \in [1, n-1]\)，计算 \(R = kG = (x_R, y_R)\) (2) \(r = x_R \bmod n\) (3) \(s = k^{-1}(H(m) + dr) \bmod n\) 签名 \((r, s)\)。 验证：\(w = s^{-1} \bmod n\)， \(u_1 = H(m) \cdot w\)，\(u_2 = r \cdot w\)， \[\text{检查 } (u_1 G + u_2 Q)_x \bmod n \stackrel{?}{=} r\] 正确性：\(u_1 G + u_2 Q = w(H(m)G + drG) = w \cdot s \cdot kG = kG = R\)。 EdDSA（确定性签名）：\(k = H(\text{prefix} \| m)\)，消除随机数 泄露风险。Ed25519 使用 Curve25519，签名 64 字节，公钥 32 字节。

🌐 实际应用

- Bitcoin/Ethereum 交易签名（ECDSA on secp256k1） - SSH 密钥推荐使用 Ed25519 - FIDO2/WebAuthn 硬件安全密钥

✏️ 例题分析与易错点

例题：为什么 EdDSA 比 ECDSA 更不容易出现实现错误？ 解答：ECDSA 需要高质量随机数 \(k\)——若 \(k\) 泄露或重用， 私钥直接暴露。EdDSA 用确定性方法 \(k=H(sk \| m)\) 从私钥 和消息派生 \(k\)，消除了随机数生成器故障的风险。 易错点：ECDSA 的签名值 \((r, s)\) 中，\(s\) 和 \(n-s\) 都是有效签名 （延展性问题）。Bitcoin 要求 \(s \leq n/2\) 来消除歧义。
